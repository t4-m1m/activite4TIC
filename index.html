Résumé du chapitre 2

Réseau(informatique) network :

Un réseau informatique est un ensemble d'ordinateurs et de périphériques autonomes connectés entre eux, situés dans une même zone géographique, qui échangent des informations numériques selon des règles définies. La mise en réseau, également appelée "Networking", consiste à utiliser des outils et des tâches pour connecter les ordinateurs afin qu'ils puissent partager des ressources.

Un réseau permet la communication de données entre différentes applications, peu importe la distance qui les sépare. Les réseaux informatiques ont évolué à partir du besoin de faire communiquer des terminaux distants avec un site central, puis entre des ordinateurs eux- mêmes. Initialement, ces communications étaient principalement axées sur le transfert de données, mais aujourd'hui, les réseaux intègrent également la voix et la vidéo.

Un réseau repose sur deux concepts fondamentaux :
• L’interconnexion : qui assure la transmission des données d'un nœud (extrémité d'une connexion) à un autre.
• La communication : qui permet l'échange de données entre les processus (les programmes en cours d'exécution) au sein du réseau informatique.
Les caractéristiques de base d’un réseau :

Les caractéristiques essentielles d'un réseau incluent sa topologie physique et logique, son débit de transmission, sa distance maximale de portée et le nombre maximum de nœuds interconnectables.


Quelques terminologies en réseau :
En réseau, vous rencontrerez des termes tels que station de : travail, nœud, paquet, réseaux homogènes, réseaux hétérogènes et débit. Comprendre ces termes vous aidera à mieux appréhender le fonctionnement et la communication au sein des réseaux informatiques.
Eléments d’un réseau :
Les éléments essentiels d'un réseau comprennent : Les éléments d'un réseau comprennent les ordinateurs connectés au réseau, les logiciels nécessaires pour accéder aux services du réseau, les supports de transmission utilisés pour le transfert des données, tels que les câbles et les connexions sans fil, ainsi que les équipements d'interconnexion, tels que les répéteurs, les concentrateurs, les commutateurs et les routeurs, qui facilitent la communication entre les appareils du réseau.

Classification des réseaux :

Les réseaux peuvent être classés selon différents critères :
Par portée (distance) :
Ils peuvent être des réseaux locaux (LAN), métropolitains (MAN) ou étendus (WAN) L'ouverture (utilisation) : En interne (intranet), en externe (extranet) ou à l'échelle mondiale (internet),
Les supports de connexion :
• Réseaux filaires : utilisant des câbles tels que le coaxial, les paires torsadées, la fibre optique et les lignes téléphoniques.
• Réseaux sans fil : basés sur des moyens tels que les ondes radio, les ondes infrarouges, les satellites et le Bluetooth.
Le type d'organisation :
- Peer to Peer : chaque ordinateur est à la fois serveur et client, permettant le partage des ressources.
- Client/serveur : la plupart des stations sont des postes clients et quelques-unes sont dédiées à des tâches spécialisées en tant que serveurs. Ce type de réseau est géré par un administrateur.

Les types principaux de topologies de réseau sont le bus, l'étoile, l'anneau et la topologie maillée. Chacune présente des caractéristiques spécifiques et le choix de la topologie dépend des besoins, des contraintes et de la taille du réseau.

Résumé du chapitre 3

Les composants matériel d'un réseau informatique :

Les composants matériels d'un réseau informatique comprennent les cartes réseau, les switches et les hubs : les cartes réseau permettent la connexion d'un ordinateur à un réseau, les switches facilitent la communication sélective entre les appareils, tandis que les hubs transmettent les données à tous les appareils connectés.

Les switches sont généralement préférés aux hubs en raison de leurs fonctionnalités avancées et de leur meilleure gestion du trafic réseau.

Le modèle OSI:

Le modèle OSI (Open Systems Interconnection) est un modèle de référence qui divise les communications en réseau en sept couches distinctes. Chaque couche a des fonctions spécifiques pour assurer la communication entre les appareils du réseau. Les sept couches du modèle OSI sont :
• Couche physique : transmission des données sur le support physique.
• Couche liaison de données : transmission des données entre les appareils partageant le même support physique.
• Couche réseau : routage des données à travers le réseau.
• Couche transport : acheminement des données de bout en bout entre les appareils.
• Couche session : établissement et gestion des sessions entre les appareils.
• Couche présentation : conversion des données dans un format compréhensible par l'appareil de destination.
• Couche application : communication entre les applications sur les appareils du réseau.

Le modèle TCP/IP:

Le modèle TCP/IP (Transmission Control Protocol/Internet Protocol) est un modèle de référence utilisé dans la conception et la mise en œuvre des réseaux informatiques modernes. Contrairement au modèle OSI, qui comprend sept couches distinctes, le modèle TCP/IP se compose de quatre couches principales :
• Couche liaison de données : responsable de la transmission des données sur le support physique.
• Couche réseau : chargée du routage des données à travers le réseau.
• Couche transport : assure l'acheminement des données de bout en bout entre les appareils.
• Couche application : facilite la communication entre les applications sur les appareils du réseau.
Le modèle TCP/IP est utilisé par les protocoles de communication réseau les plus répandus, tels que le protocole Internet (IP), le protocole de contrôle de transmission (TCP) et le protocole de datagramme utilisateur (UDP). Il sert de base pour la conception et la mise en place de réseaux d'entreprise, d'Internet et de réseaux sans fil.

L'adressage:

l'adressage IP est un élément fondamental des réseaux informatiques qui permet l'identification unique des dispositifs et facilite la transmission des données entre eux. IPV4 et IPV6 sont deux versions d'adressage IP couramment utilisées, chacune ayant ses propres spécifications et représentations.

L'adresse IPV4:

l'adresse IPV4 est divisée en deux parties : l'adresse réseau et l'adresse hôte.
Les adresses réseau ont tous les bits de la partie hôte définis à 0, les adresses de diffusion ont tous les bits de la partie hôte définis à 1, et les adresses hôtes sont attribuées aux machines situées sur le réseau entre l'adresse réseau et l'adresse de diffusion. Le masque de réseau et de sous-réseaux est utilisé pour déterminer les parties réseau et hôte d'une adresse IP.

Résumé du chapitre 5

Dans ce chapitre, Git et GitHub sont définis pour éviter toute confusion, et la popularité de Git ainsi que l'importance des systèmes de contrôle de version sont discutées. Git est un logiciel de versionnèrent créé en 2005 par Linus Torvalds, le créateur de Linux, et il est le leader incontesté parmi les systèmes de contrôle de version. Le logiciel de contrôle de version permet de garder une trace des modifications apportées à un projet pour identifier rapidement et revenir à une version antérieure en cas de problèmes. Un tel logiciel est essentiel pour une gestion de projet et une collaboration efficace.

Deux modèles de systèmes de contrôle de version sont décrits : le modèle centralisé et le modèle décentralisé, également appelé modèle distribué. Dans le modèle centralisé, le code source du projet est hébergé sur un serveur distant central, et les utilisateurs doivent s'y connecter pour travailler sur le code. Dans le modèle distribué, chaque utilisateur télécharge et héberge l'ensemble du code source du projet sur sa machine, ce qui lui permet de travailler sans connexion constante à un serveur central. Git a popularisé le modèle distribué, offrant simplicité, flexibilité, sécurité et la possibilité d'utiliser la copie du projet d'un utilisateur comme sauvegarde en cas de corruption du serveur.

1) Installation de Git

La manière la plus facile d'installer Git est de télécharger la dernière version depuis le site officiel http://git-scm.com/downloads, d'ouvrir le fichier téléchargé et de suivre les instructions à l'écran en laissant toutes les valeurs par défaut.

2) Configuration de Git

Une fois que Git est installé, nous allons configurer le logiciel pour enregistrer certaines données afin que nous n'ayons pas à les fournir à nouveau plus tard. Nous allons, par exemple, fournir un nom d'utilisateur et une adresse e-mail que Git utilisera plus tard. Pour cela, nous utiliserons notre première commande Git, qui est la commande git config. Cette commande nous permet de visualiser et de modifier les variables de configuration qui contrôlent tous les aspects de l'apparence et du comportement de Git.

Nous passerons également une option --global à notre commande. Les options permettent de personnaliser le comportement par défaut de certaines commandes. Ici, l'option --global nous permettra d'indiquer à Git que le nom d'utilisateur et l'adresse e-mail fournis doivent être utilisés globalement (c'est-à-dire pour n'importe quel projet Git).
Nous entrerons donc les commandes suivantes : git config --global user.name "Bouchekir Redouane" et git config --global user.email redouane.bouchekir@gmail.com l'une après l'autre pour fournir un nom et une adresse e-mail. Bien sûr, utilisez votre propre nom et adresse e-mail.
Pour vous assurer que vos informations ont été enregistrées, vous pouvez entrer git config user.name et git config user.email. Les informations saisies doivent être retournées.

3) Démarrage d'un dépôt Git

Un "dépôt" dans Git fait référence à l'importation et la copie de tous les fichiers d'un projet dans Git. Il existe deux façons de créer un dépôt Git : importer un répertoire existant dans Git ou cloner un dépôt Git existant.

4) Gestion des informations dans Git

Git gère les données d'une manière différente de la plupart des autres systèmes de contrôle de version. Git considère les données comme un flux de snapshots. Chaque fois qu'un état du projet est validé dans Git, il prend un instantané du contenu de l'espace de travail et enregistre une référence à l'instantané pour qu'il puisse être accédé ultérieurement. Chaque instantané est stocké dans une base de données locale sur notre machine.
Avoir un historique complet du projet localement signifie que la plupart des opérations Git peuvent être effectuées localement sans avoir à être connecté à un serveur distant central. Cela rend les opérations plus rapides et le travail plus agréable en général.

5) États des fichiers

Comment Git suit-il les modifications apportées aux fichiers d'un projet ? Pour comprendre cela, il est essentiel de savoir qu'un fichier peut avoir deux états majeurs dans Git : suivi ou non suivi.
Un fichier est "suivi" s'il faisait partie de la dernière capture d'écran réalisée par Git, ce qui signifie qu'il est enregistré dans la base de données. Tout fichier qui n'a pas été indexé et qui n'était pas inclus dans la dernière capture d'écran est "non suivi".
Lorsque l'on commence un référentiel Git en important un répertoire existant de notre machine, tous les fichiers sont initialement non suivis. Par conséquent, nous devons dire à Git de les indexer et de les valider (les enregistrer dans la base de données). Lorsque l'on clone un référentiel Git existant, c'est différent car nous copions l'historique complet du projet et les fichiers sont déjà suivis par défaut.
De plus, chaque fichier suivi peut avoir l'un des trois états suivants :
 Modifié ("modified")
 Stagé ("staged")
 Validé ("committed")
Lorsque l'on commence un référentiel Git à partir d'un référentiel local, nous demandons à Git de valider tous les fichiers du projet. Un fichier est "validé" lorsqu'il est enregistré dans la base de données locale. Lorsque l'on clone un référentiel existant, les fichiers sont enregistrés dans la base de données par défaut.
Lorsque l'on travaille sur un projet, nous pouvons ajouter de nouveaux fichiers ou modifier ceux qui existent. Les fichiers modifiés sont considérés comme "modifiés" par Git jusqu'à ce qu'ils soient stagés. Nous stagons un fichier en disant à Git que le fichier modifié ou nouveau doit faire partie de la prochaine capture d'écran dans sa version actuelle. Enfin, lorsque nous demandons à Git de prendre la capture d'écran, c'est-à-dire d'enregistrer l'état actuel du projet (c'est-à-dire tous les fichiers stagés et non modifiés), les fichiers de la capture d'écran sont à nouveau considérés comme "validés" et le cycle recommence.

6) Zones de travail

Les états des fichiers sont liés aux zones de travail dans Git. Selon leur état, un fichier peut apparaître dans une zone de travail spécifique. Chaque projet Git est composé de trois sections :
le répertoire de travail, l'aire de préparation et le répertoire Git.
Le répertoire de travail ou "arborescence de travail" correspond à une extraction unique ("checkout") d'une version de projet. Les fichiers sont extraits de la base de données compressée située dans le répertoire Git et sont placés sur le disque pour être utilisés ou modifiés.
L'aire de préparation ou "index" correspond à un simple fichier, généralement situé dans le répertoire Git, qui stocke des informations sur ce qui fera partie de la prochaine capture d'écran ou "commit".
Le répertoire Git est l'endroit où Git stocke les métadonnées et la base de données d'objets de votre projet. C'est la partie principale ou le cœur de Git.
Le processus de flux de travail est le suivant : nous travaillerons sur nos fichiers dans le répertoire de travail. Lorsque nous modifions ou créons un fichier, nous pouvons ensuite choisir de le mettre en préparation. Tant qu'un fichier n'est pas mis en préparation, il a l'état modifié ou est non suivi s'il s'agit d'un nouveau fichier. Une fois qu'il est mis en préparation, c'est-à-dire que son nom est ajouté à l'aire de préparation, il a l'état mis en préparation. Enfin, nous validerons ("commit") la version préparée de nos fichiers pour les ajouter au répertoire Git.

7) Créer un dépôt Git à partir d'un répertoire existant

Lorsque nous commençons avec Git, nous avons souvent déjà des projets en cours stockés localement sur notre machine ou sur un serveur distant pour lesquels nous aimerions implémenter un système de contrôle de version.
Dans ce cas, nous pouvons importer toutes les ressources d'un projet dans Git. Pour le reste de cette leçon, je vais créer un répertoire "git-projet" situé sur mon bureau et contenant deux fichiers texte vides "file1.txt" et "README.txt". Ce répertoire servira de base pour les exemples qui suivront (ce sera le répertoire importé).
Remarque : Dans cmd, les commandes :
 cd <chemin> : vous permet de vous positionner à un emplacement dans le PC, c'est-à- dire de vous placer dans un répertoire.
 dir : vous permet d'afficher la liste des dossiers et des fichiers dans l'emplacement actuel.
Pour initialiser un dépôt Git, nous utilisons ensuite la commande git init. Cela crée un sous- répertoire .git contenant un ensemble de fichiers qui permettront à un dépôt Git de fonctionner. Lorsque nous utilisons git init, Git nous renvoie un message nous informant que le dépôt Git a été initialisé et qu'il est vide. C'est parfaitement normal car nous n'avons pas encore versionné de fichiers (nous n'avons pas ajouté de fichiers du répertoire à la base de données).
Nous pouvons utiliser la commande "git status" ici pour déterminer l'état des fichiers dans notre répertoire. Cette commande est extrêmement utile et c'est celle que j'utilise personnellement le plus souvent.
Ici, "git status" nous informe que notre projet a deux fichiers qui ne sont pas sous contrôle de version ("non suivis") et ce sont les fichiers "README.txt" et "file1.txt". La prochaine étape sera de mettre en stage nos fichiers afin qu'ils puissent être validés, c'est-à- dire ajoutés à la base de données et que nous puissions ainsi avoir une première historique de version.
Pour mettre en stage des fichiers, nous utilisons la commande "git add". Nous pouvons lui passer un nom de fichier pour mettre en stage le fichier en question, le nom d'un répertoire pour mettre en stage tous les fichiers du répertoire en une seule fois, ou un "fileglob" pour ajouter tous les fichiers correspondant au modèle fourni.
Les "fileglobs" utilisent les extensions de chemin de fichier. Cela signifie que certains caractères comme * et ? auront une signification spéciale et nous permettront de créer des modèles de correspondance. Le caractère *.

La deuxième façon de démarrer un dépôt Git est de cloner localement un dépôt Git existant. Pour ce faire, nous utilisons la commande "git clone". Créez un dépôt sur GitHub. 9) Ajouter ou modifier des fichiers dans un projet et mettre à jour notre dépôt Git

À ce stade, nous avons d'une part notre projet contenant un ensemble de fichiers et de ressources sur lesquels nous travaillons, et d'autre part, un dépôt Git qui est utilisé pour gérer les différentes versions de notre projet. Au fur et à mesure que nous continuons à travailler sur notre projet, nous ajouterons, modifierons et même supprimerons des fichiers. Nous indiquerons toutes ces modifications à Git afin qu'il puisse garder une historique des versions que nous pourrons consulter ultérieurement pour revenir à un état précédent du projet (en cas de bug, par exemple, ou si une modification ne donne pas le résultat souhaité). Chaque fois que nous voulons enregistrer une modification de fichier ou ajouter un fichier au dépôt Git, nous devrons utiliser les commandes "git add" et "git commit" comme nous l'avons fait dans la leçon précédente. Ici, je attire votre attention sur un point important : le commit (validation/enregistrement dans la base de données) d'un fichier sera basé sur l'état de ce fichier au moment de "git add". Cela signifie que si vous effectuez une commande "git add" sur un fichier, puis modifiez à nouveau le fichier et effectuez ensuite un "git commit", Il s'agit du fichier dans son état au moment du dernier git add qui sera validé, et les dernières modifications ne seront pas enregistrées dans le dépôt Git. Si vous souhaitez toujours enregistrer la dernière version d'un fichier, assurez-vous de toujours effectuer un git add juste avant un git commit. Pour ajouter les fichiers modifiés qui sont déjà suivis dans la zone d'index et les valider, vous pouvez également utiliser git commit avec une option -a comme ceci : git commit -a. Cela vous dispense d'avoir à taper git add.

10) Consulter l'historique des modifications Git

La façon la plus facile de consulter l'historique des modifications Git est d'utiliser la commande git log. Cette commande affiche la liste des commits effectués du plus récent au plus ancien. Par défaut, chaque commit est affiché avec son checksum SHA-1, le nom et l'adresse e-mail de l'auteur, la date et le message de commit.

11) Annuler les modifications apportées à un fichier

L'un des principaux avantages de l'utilisation d'un logiciel de contrôle de version est la capacité à "revenir en arrière", c'est-à-dire à pouvoir retourner à un état précédemment enregistré d'un projet. Après un commit, nous continuerons à travailler sur nos fichiers et à les modifier. Parfois, certaines modifications ne donneront pas les résultats escomptés, et nous voudrons revenir à l'état du fichier de la dernière capture Git (c'est-à-dire le dernier état enregistré). Nous pouvons le faire avec la commande générale git checkout -- nomdufichier ou la nouvelle commande spécialisée git restore.

2. Qu'est-ce qu'une branche ?
 Une branche dans Git est un pointeur vers un commit, elle permet de créer une copie virtuelle du projet pour y travailler sans affecter la branche principale. Git utilise des pointeurs simples pour représenter les branches, facilitant ainsi leur création et leur fusion.
 Lorsqu'un commit est effectué, Git stocke les informations dans un objet de commit, qui est ensuite référencé par la branche.
 Les branches sont un moyen efficace de travailler sur différentes fonctionnalités du projet en parallèle.
